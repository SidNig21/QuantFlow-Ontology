# WO-003 — Full schema v0.2: every type, every legal transition, enforced by gates

> **Builder: this file plus the two documents it names are your entire context. Do not improvise beyond it.**

## Objective

Expand the WO-001 spike slice into the complete frozen ontology (schema v0.2) with legal-transition tables that **generate** conformance tests, so that every illegal state transition is a failing test and schema drift is impossible to merge.

## Context pack (read in this order, nothing else)

1. `START_HERE.md` — mission and the One Rule.
2. `docs/ONTOLOGY_SCHEMA.md` — **the specification you are implementing.** Every object, property, link, action, and the §State machines section.
3. `qf-kernel-schema/` — the WO-001 package you are extending. Its patterns (`defineObject/Link/Action`, pure generators, goldens) are settled; keep them.

## Deliverables (all inside `qf-kernel-schema/`)

1. **Full schema** (`src/schema.ts`) — every object in `docs/ONTOLOGY_SCHEMA.md`: domain plane (`competitor`, `event`, `market`, `odds_series`, `result`), research plane (`hypothesis`, `strategy`, `ticket`, `dataset`, `run`, `artifact`, `evaluation`), ops plane (`workspace`, `agent_definition`, `agent_session`, `task`, `tool`, `execution_environment`, `connection`). Every link and action in the doc. Descriptions written for an agent reader — what the thing IS in betting-research terms. **The schema doc is authoritative; if it is silent on a property's type, choose the obvious one and list it in your report.**

2. **Multi-source links.** `defineLink` currently takes one `from` and one `to`. The schema needs `tests` (run **or** strategy → hypothesis), `uses` (run → dataset|strategy|tool), `produces` (run|agent_session → dataset|artifact), `derived_from` (dataset|artifact|strategy → same). Extend `from`/`to` to accept an object **or a non-empty array** of objects, preserving declaration order in all generated output.

3. **Transition tables** (`src/transitions.ts`) — the §State machines tables as data, beside the types:
   ```ts
   export const transitions = {
     run: { queued: ["running"], running: ["succeeded","failed","cancelled"],
            succeeded: [], failed: [], cancelled: [] },
     // hypothesis, ticket, event, agent_session per the doc
   } as const;
   ```
   Every state in a type's status enum must appear as a key — a missing key is a lint failure naming the offender. Terminal states map to `[]`.

4. **Command/event split** (`src/commands.ts`) — for each stateful type, the commands that may cause a transition and the event each emits on success (`start_run` → `run.started`). Commands are rejectable intents; events are facts. **No Receipt type** — the event log is the receipt log.

5. **Generated conformance tests** (`src/generate/conformance.ts` → `golden/conformance.test.ts`): for every type, for every state, emit one test per **illegal** target state asserting the transition is rejected, plus one test per legal transition asserting it is accepted. The generated file is committed as a golden and runs as part of `bun test`. A hand-written test file for transitions is a defect — these must be generated from the tables.

6. **Transition validator** (`src/validate.ts`) — the pure function the tests exercise: `canTransition(type, from, to): boolean` and `assertTransition(type, from, to): void` throwing a message naming type, from, and to. This is the function the Kernel will call at the command layer in WO-005.

7. **Lint extension** (`src/define.ts`) — add, each failing with a message naming the offender:
   - every stateful object's status enum is covered by a transition table (and vice versa — no orphan table keys);
   - no duplicate object/link/action names;
   - link endpoints reference objects that exist in the schema.

8. **`lifecycle` is type metadata, not row data — order defect from WO-001, corrected here.** Remove the per-row `lifecycle` column from object tables. Emit instead a single generated `schema_meta` table (`type_name TEXT PRIMARY KEY, kind TEXT, lifecycle TEXT NOT NULL CHECK (lifecycle IN ('experimental','active')), description TEXT`) populated by the migration. Rationale: lifecycle governs the *type*, and stamping it on every ticket row conflates schema governance with data.

9. **Generator corrections** (also from WO-001 seam inspection):
   - `links.kind` gets `CHECK (kind IN (...))` listing every defined link name.
   - `qf_<object>_get` and `qf_<object>_query` must have **distinct** descriptions (get = fetch one by id; query = list with filters), both still derived from the object's description. Agents route on these strings; identical text is a Misnomer defect.

10. **Regenerated goldens** — `migration.sql`, `tools.json`, `ONTOLOGY.md`, `conformance.test.ts`, all committed.

11. **A repo qa gate** — `qa/run.ts` gains a `schema` gate that runs the package's tests and fails the build on any lint, golden, determinism, or conformance failure. It must appear in `bun qa/run.ts --list` and pass in `--all`.

### Correction — verification was premature (2026-07-18)

**CI failed the merge commit that recorded this PASS.** Honest record of what happened:

`qf-kernel-schema/` is a separate package with its own lockfile. CI installs at the repo root only, so the sub-package's dependencies were never present on a fresh checkout and the `schema` gate died on `Cannot find package 'zod'` — 120 tests collected instead of 137, 2 files erroring on import.

**Both the builder and the verifier masked it identically:** each ran `bun install` inside the package before running the gate, so the gate found dependencies a clean clone does not have. The order's acceptance gates listed `cd qf-kernel-schema && bun install` before `bun qa/run.ts --all`, which made the masking structural rather than accidental — that is an order defect, mine.

**Fix (commit follows):** the `schema` gate now runs `bun install --frozen-lockfile` in the sub-package itself before testing, so it behaves identically on a fresh clone and on a warm machine, and cannot drift from the committed lockfile. Verified by deleting `qf-kernel-schema/node_modules` and running the gate cold: 137 pass, 0 fail.

**Standing rule this establishes:** a gate may not depend on ambient machine state. Any future sub-package gate installs its own dependencies. Verification of a gate must be done from a cold state, not after a convenience install.

The schema work itself was never in question — 137/137 with dependencies present, and every seam-inspection finding above still holds. The defect was in the gate's wiring, not the ontology.
