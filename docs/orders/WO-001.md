# WO-001 — Codegen spike: schema → migrations + MCP tools + docs

> **Builder: this file is your entire context. Do not improvise beyond it.**

## Objective
Prove the L0→L3 bet: one Zod schema definition generates SQL migrations, MCP tool definitions, and markdown docs — so schema drift becomes a failing test, never a runtime surprise.

## Context (all you need)
QuantFlow is a Linux-first research console whose Kernel is a typed ontology (objects, links, actions) in SQLite. This spike builds the **schema-definition package** standalone — no Electron, no app, no database connection. Pure TypeScript + Bun. It will later be dropped into the main repo unchanged.

## Deliverables — a Bun package `qf-kernel-schema/`
1. **Schema core** (`src/define.ts`): tiny helpers `defineObject / defineLink / defineAction` wrapping Zod, each requiring: `name` (snake_case singular), `description` (non-empty — build must fail without it), `lifecycle: "experimental" | "active"`, properties as Zod fields each with `.describe()` (non-empty enforced), and for links: `from`/`to` object refs.
2. **Schema instance** (`src/schema.ts`) — exactly this slice, nothing more:
   - Objects: `hypothesis` (claim: string, success_criteria: string, sources: string[], status: enum open|supported|rejected|inconclusive) · `event` (sport: enum ufc|tennis|football, starts_at: ISO datetime, status: enum scheduled|live|settled|void, competition: string) · `market` (kind: enum moneyline|spread|total|prop, params: JSON, sides: string[], correlation_group: string nullable) · `ticket` (kind: enum single|parlay, legs: JSON array, combined_price: number, stake: number, grade: enum pending|win|loss|push|void)
   - Links: `offered_on` (market→event) · `has_leg` (ticket→market) · `tests` (ticket→hypothesis)
   - Actions: `create_hypothesis` (input: claim, success_criteria, sources?) · `submit_ticket` (input: kind, legs, combined_price, stake)
   - Every description written for an AI agent reader: what the thing IS in betting-research terms, one or two sentences. No placeholder text.
3. **Generators** (each a pure function, output deterministic):
   - `generate/sql.ts` → one migration file: object tables (id TEXT pk, created_at, lifecycle + typed columns), a single `links` table (id, kind, from_id, to_id, created_at), comments from descriptions.
   - `generate/mcp.ts` → JSON array of MCP tool definitions: per object `qf_<name>_get` / `qf_<name>_query`, per action `qf_<action_name>`; `description` fields taken verbatim from schema descriptions; `inputSchema` via Zod→JSON-Schema conversion.
   - `generate/docs.ts` → `ONTOLOGY.md`: object/link/action catalog with properties and descriptions, in schema order.
4. **Goldens** (`golden/migration.sql`, `golden/tools.json`, `golden/ONTOLOGY.md`): committed generated output.
5. **Tests** (`bun test`):
   - Golden tests: regenerating each output matches the committed golden byte-for-byte.
   - Lint tests: a schema entry missing a description or lifecycle fails with a readable error naming the offender.
   - Determinism: two consecutive generations are identical.

## Contract
- Bun + TypeScript strict. Zod v4. Zod→JSON-Schema via Zod's own conversion if available in v4, else `zod-to-json-schema`. **No other runtime dependencies.**
- Generators never read the filesystem for input — schema in, string out.
- Do not add objects/properties beyond the slice. Do not "helpfully" build a database layer, a server, or CLI wrappers.
- All names snake_case; tool names prefixed `qf_`.

## Acceptance gates (run these; paste full output in your report)
```bash

bun test          # all green, includes golden + lint + determinism tests
bun run generate  # package.json script; regenerates into golden/ with zero git diff
```
