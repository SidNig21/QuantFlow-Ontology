## Deliverables

1. **`tools/runtime-proof/`** — a standalone Bun + TypeScript package containing the whole proof harness. It is a spike: no Electron, no canvas, no Kernel, no SQLite, and nothing copied from the predecessor repo.

2. **The chain, minimally wired:**
   - an **AgentOS session** started directly against `@rivet-dev/agentos`, which owns the public session lifecycle and is the source of the session ID;
   - a **custom ACP agent** (`@agentclientprotocol/sdk`, v1.2.1) as the bridge — the ACP layer must carry the *same* session ID it was given, never mint its own;
   - **`ToolLoopAgent`** from the Vercel AI SDK (`ai`, v7.x), owning the model/tool loop inside that session.

   **Drive it with a mock model, not a live one.** Use the mock provider from `ai/test` (e.g. `MockLanguageModelV2` + `simulateReadableStream`) scripted to emit: some text chunks → a tool call → more text. This order proves *plumbing* — session identity, loop wiring, streaming, cancellation — none of which needs a real model, and all of which is more reliable to assert against a deterministic script. **No API key is required for any of P1–P4, and the qa gate must run fully without one.** A live-model smoke test is optional and out of scope; do not add one.

3. **One harmless tool** — a pure function the model can call, e.g. `echo_upper(text: string) -> string`. No filesystem, no network, no side effects.

4. **Incremental streaming** — assistant output must arrive as a stream of chunks, not one final blob. Prove it: record chunk arrival timestamps and assert more than one distinct chunk.

5. **Cancel mid-turn** — cancel a run while the model is streaming; assert the stream stops, the session reaches a terminal state, and **no orphan process or timer remains**.

6. **The proof assertions** (this is the order's whole point — each must be a runnable test, not prose):
   - **P1 · One session ID.** The ID reported by AgentOS, the ID carried by the ACP layer, and the ID attached to the tool-loop turn are **identical**. Assert equality on the actual runtime values, not on a variable you passed around by hand.
   - **P2 · No second server.** Exactly one process listens. Enumerate listening sockets (or the host's own registry) before and after starting a session and assert no additional listener appeared. A second Eve-style server process fails this order.
   - **P3 · Tool call inside the session.** The tool is invoked by the model loop and its result reaches the final assistant message.
   - **P4 · Cancel is clean.** Per deliverable 5.

7. **A repo qa gate** — `qa/run.ts` gains a `runtime-proof` gate that runs the proof suite. It must appear in `--list` and pass in `--all`. Two hard requirements, both learned from WO-003's CI failure:
   - **The gate installs its own dependencies** (`bun install --frozen-lockfile` in each package it runs) — it must pass on a fresh clone with no `node_modules` anywhere. Verify this yourself by deleting `node_modules` and running `bun qa/run.ts --all` cold before you report.
   - The gate runs the full suite with **no API key present** — that is the normal case, not a degraded one. Nothing may skip for want of a credential.

**The verdict is sound. P1–P4 hold, and the proof is genuine, not tautological:**

- **P1 is a real cross-boundary assertion.** The AgentOS-side ID comes from `os.createSession()`; the ACP and tool-loop IDs are read from a receipt the **guest process writes** and the host reads back via `os.readFile`. Not a closed-over variable, which is exactly what the order demanded. The session UUID differs on every run (`6ad011b0…` in the builder's report, `72998927…` in mine) — regenerated, not fixtured.
- **P3/P4 exercise the real path** — tool result reaches the assistant message, multi-chunk streaming confirmed, cancel yields `cancelled` with one chunk before drain.
- **No orphan processes.** The order asked for this and P4 asserts only listeners, so I tested it directly: ran the suite and counted surviving agent processes — **before = after, zero leaked**, nothing alive under the packed agent.

### Finding 1 — session-ID directionality is inverted from the docs (architectural, not a defect)

`acp-main.ts:114` — the **ACP agent mints the session ID** (`crypto.randomUUID()` in `newSession`); **AgentOS adopts it**. `BLUEPRINT.md` and this order both asserted the opposite ("AgentOS … is the source of the session ID"). That was my error: ACP's `session/new` returns the ID by protocol, so the client cannot impose one.

**The invariant that matters is unaffected** — one ID, no second identity, AgentOS still owns create/cancel/destroy. But the direction is load-bearing for WO-005: the Kernel's `agent_session` identity is **adopted from the guest**, not minted by the Kernel. Corrected in `BLUEPRINT.md` in the merge commit. The builder surfaced this in §7 rather than quietly matching the doc — correct behavior.

### Finding 2 — what P2 actually proves (carry-forward, not rework)

`AgentOs.create` embeds AgentOS in-process and opens **zero** listeners, so "no new listeners" passes without discriminating much. The substantive property still holds and is better than the order asked: the agent is a **stdio child process**, not a socket server — no second server, and no second session registry. Noted so nobody later reads P2 as proving more than it does. The real listener/soak pressure belongs to WO-006's proof bar.

### Finding 3 — P4's assertion is softer than its intent (carry-forward)

`proof.test.ts:70` accepts `["cancelled", "end_turn"]`. A run where cancel lost the timing race would pass **without having tested cancellation**. Observed behavior is correct (`cancelled`, 1 chunk before drain) and the timing margin is generous, so this is not rework — but WO-006's cancel path must assert `cancelled` strictly, and additionally assert no orphan **process**, which P4 checks only for listeners.
