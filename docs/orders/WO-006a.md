# WO-006a — Creation commands: the Kernel can publish an Artifact

> **Builder: this file plus the documents it names are your entire context. Do not improvise beyond it.**

## Objective

Give the Kernel a **creation-command** path — a command that brings an object into existence through the command/event layer — and make `publish_artifact` the first one, with content-addressing that is computed and enforced rather than asserted.

## Why this order exists (measured, not assumed)

WO-006's Law D acceptance path requires *"create an Artifact via a Kernel action."* **Kernel v0 cannot do that.** Measured on `QuantFlow` @ `a80e763`:

- `execute()` (`packages/qf-kernel/src/execute.ts:123`) handles **state transitions only**: it reads a current state, calls `assertTransition`, then issues `UPDATE <table> SET <field>`. It has no path that inserts a row.
- Object creation exists only as `insertRun` and `insertAgentSession` (`src/insert.ts`), which are direct inserts.
- The `artifact` table exists in the generated migration and has **no status column** — artifacts are immutable, so the transition machinery does not and should not apply to them.
- `publish_artifact` exists as a schema action (`qf-kernel-schema/src/schema.ts:625`) with input `{ kind, content_hash, storage_ref }`, and nothing consumes it.

So WO-006 would have hit this wall on day one. This order clears it while the work is still cheap and headless.

## Deliverables

1. **A creation-command class in the Kernel.** `execute()` currently assumes every command is a transition. Extend it so a command may be a **creation** instead, without special-casing `artifact` by name. A creation command must, in **one transaction**:
   - require trace context (reject with the existing `MissingTraceError` if absent — same rule as transitions);
   - insert the new row;
   - append exactly one event to the same append-only `events` table;
   - return the created state.

   Direct inserts that bypass the event log are a Law B violation. `insertRun` / `insertAgentSession` may remain as test fixtures, but **the artifact path may not use them.**

2. **`publish_artifact` wired as the first creation command**, consuming the existing schema action. Do not invent a new action name; do not edit `schema.ts`'s action surface to make this easier.

3. **Content-addressing that is enforced, not claimed.** The schema calls an artifact *"immutable, content-addressed"*, and today nothing checks that. The command takes the **bytes** (a `Uint8Array`, or a path the Kernel reads):
   - the Kernel **computes** `content_hash` itself;
   - if the caller also supplies a `content_hash` that disagrees with the computed one, **reject** with a typed error and write nothing;
   - the artifact's `id` is derived from the content hash, so identical bytes are the same artifact.

4. **Republish is idempotent.** Publishing identical bytes twice must **not** create a second artifact row. Decide and state in your report whether the second publish appends a second event or is a no-op — either is defensible, but the row count must stay 1 and the behaviour must be tested both ways.

5. **The commands lint must survive.** WO-005's bidirectional lint (every command name is a real schema action; every legal transition has a command) currently assumes commands are transitions. Extend it to cover creation commands **without weakening either direction** — a creation command still must name a real schema action, and no legal transition may lose its command. Adding a fourth catalog must remain impossible.

6. **Replay covers artifacts.** WO-005 replays `run` state from the event log. Extend replay so artifact **existence** is rebuilt from events alone and asserted equal to the live table — same standard: two independent derivations, never a value compared to itself.

7. **Debt #0 — the doc↔code action-surface gate.** `docs/ROADMAP.md` assigns this to *"the next order touching the schema surface."* That is this order. Add a gate asserting the action list in `docs/ONTOLOGY_SCHEMA.md` §Actions equals the actions exported by `schema.ts`, failing with the names that differ in each direction. WO-005 shipped 25 actions against a 13-action doc and only a human reading two lists caught it. **You may not edit `docs/ONTOLOGY_SCHEMA.md` to make the gate pass** — if the two disagree, make the gate fail, report the difference, and stop. Reconciling the doc is the architect's call by order.

### Defects (numbered, all measured)

**D1 — Idempotent republish silently swallows conflicting metadata.** Verifier's probe, run against `fd49d2a`: publish bytes as `report` @ `file:///a`, republish the *same bytes* as `strategy_spec` @ `file:///b` → returns `report` @ `file:///a`, no error, no event. The caller believes a fact the Kernel does not hold. Content-addressing makes identical bytes idempotent; it does not license lying about `kind`/`storage_ref`. **Fix:** on an existing-row hit in `publishArtifact`, compare `kind` and `storage_ref` against the existing row; mismatch → new typed error (e.g. `ArtifactMetadataConflictError`), no write, no event. Identical metadata stays a no-op. **Prove it can fail:** add the test; then temporarily disable the comparison, show the new test red, restore, show green. Paste both. *Attribution: architect — the order said "idempotent republish (row count stays 1)" and never specified conflict behavior. The builder's reading was defensible.*

**D2 — Artifact replay echoes its query key as the rebuilt `id` (decorative-assertion pattern, third occurrence).** `replay.ts`: `rebuilt.id = artifactId` — the function's own argument, also used in `WHERE id = ?`. The `live.id !== rebuilt.id` comparison cannot fail if the row exists. The other three legs (`kind`, `content_hash`, `storage_ref`) are honest — rebuilt from event payloads — so this is one dead leg, not a dead assertion. But the dead leg is exactly where the identity rule (`id` = content hash) should be asserted. **Fix:** `rebuilt.id = payload.content_hash` — event-derived, so `live.id`, the requested id, and the event's hash become three independent values. **Prove it can fail:** in a test, `UPDATE events SET payload` to carry a wrong `content_hash`, show replay throw, restore, show green. Paste both.

**D3 — Creation catalog and creation handlers are two registries with no join.** `creationCommands` (catalog) and `executeCreation`'s `if (cmd.action === "publish_artifact")` (handler) are joined by nothing. A catalog entry without a handler stays green until runtime throws. **Fix:** one dispatch table — `Record<action, handler>` (or a `Map` built beside the catalog) — plus a test asserting every `creationCommands` entry has a handler. **Prove it can fail:** add a bait catalog entry with no handler, show the test red, remove, show green. Paste both.

**D4 — The receipt log has three writers.** `appendEvent` in `execute.ts` and `create.ts` are byte-identical copies; `insert.ts` carries a third `INSERT INTO events`. The event log is the Kernel's One Rule; one writer is not polish. **Fix:** one exported `appendEvent` in one module; all three call sites use it. **Prove it:** `grep -rn "INSERT INTO events" packages/qf-kernel/src` returns exactly one occurrence; paste it.

Fix D1–D4 only; nothing else. Same branch, `wo-006a`. D1 and D2 are integrity, D3 and D4 are structure the review correctly refused to let ride on a Kernel whose law is the event log.
