# WO-009 — Datasets I: bootstrap ingestion
status: open
assignee: builder
depends: none (parallel-eligible; Kernel + `publish_artifact` / run lifecycle already shipped)
created: 2026-07-20

## Objective

Land the first real **data plane**: an `ingestion` Run that materializes a content-hashed **Parquet** snapshot, registers a Kernel `dataset` pointer (`as_of` + `coverage`), and proves DuckDB can read via that pointer — **without** stuffing bulk rows into SQLite.

## Why this order exists

v0.5’s defining workflow needs datasets before backtests. Schema already defines `dataset`, `register_dataset_version`, `produces` / `derived_from`, and run `kind: ingestion`. The Kernel today creates artifacts; datasets are still a doc/codegen surface. This order makes the pointer + bytes contract real and falsifiable.

## Context pack

1. `docs/ROADMAP.md` §WO-009 (binding gates)
2. `docs/ONTOLOGY_SCHEMA.md` — `dataset`, `run.kind`, links `produces` / `derived_from` / `uses`
3. `qf-kernel-schema` — `register_dataset_version` action + `dataset` object (measure whether Kernel `execute` handles it; if not, implement creation path like `publish_artifact`)
4. Standing rules: cold-state, gate-falsification, Law E (Kernel sole writer), no credentials in repo

## Deliverables

**0. Measure first — stop-or-proceed.**
Confirm whether `register_dataset_version` is implemented in `packages/qf-kernel` (handler + tests). If missing, add it in this order (creation command + event + live row + replay) before ingestion orchestration. Paste the measured gap.

**1. One bootstrap source (not three).**
Implement **one** honest ingestion path end-to-end (pick the smallest that yields real rows):
- tennis-data.co.uk archive, **or**
- a **committed fixture** derived from a public archive (preferred for CI/cold — document provenance; no API keys).

Optional second source only if the first path is green and extract-first still holds. Do **not** require live Kaggle/NFL network in CI.

**2. Parquet + content hash.**
Ingestion writes Parquet under a durable local store (path recorded on the dataset / `storage_ref`-style field — measure schema). `content_hash` is over the Parquet bytes (or documented canonical set). **Falsify:** mutate one byte → different hash; identical bytes → identical hash.

**3. Kernel registration + lineage.**
Successful run: `run` (`kind: ingestion`) reaches `succeeded`; `dataset` row exists with `as_of` + `coverage`; `produces` (and `derived_from` if a parent exists) traversable. Failed/truncated source → run `failed`, **zero** partial dataset rows (and no orphan Parquet claimed by Kernel).

**4. DuckDB read proof.**
A small host/test harness opens the dataset pointer and runs a `SELECT count(*)` (or equivalent) via DuckDB against the Parquet path. Paste row count. **Gate (or package test):** fail if any ingestion path inserts bulk fact rows into `kernel.db` (pointer + metadata only).

**5. Standing gates.**
`bun qa/run.ts` static set green from cold. Extract-first if touching Electron shell JS (prefer not to — this order is Kernel + data plane). `agent-host.ts` / Hermes path must not regress.

## Contract

- No bulk time-series in SQLite.
- No credentials in repo, fixtures, or logs.
- Do not weaken existing Kernel / agent-path falsifications.
- Schema edits only if measured necessary — prefer implementing the already-declared `register_dataset_version`.
- Prefer a package under `packages/` or `tools/` over growing Electron.

## Acceptance gates — builder-run (paste unedited)

1. Every static gate (PROTOCOL rule).
2. Deliverable 0 measurement (handler exists or was added).
3. Ingestion run → `succeeded` + dataset id + `content_hash` + `as_of`/`coverage`.
4. Hash falsify: one-byte mutate → hash changes.
5. DuckDB count proof (paste).
6. Failure honesty: bad/truncated input → `failed`, zero dataset rows.
7. Proof of no bulk fact tables in `kernel.db` for that run (gate or sqlite inspect).

**Never `bun qa/run.ts --all` as the builder’s only proof if you add a new package gate — add the gate and run it; verifier re-runs cold.**
