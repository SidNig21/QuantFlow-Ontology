# WO-005 — Kernel v0: the only place truth lives

> **Builder: this file plus the documents it names are your entire context. Do not improvise beyond it.**

## Objective

Stand up the Kernel: a SQLite database created **from the generated migration**, with a command layer that validates every state change against the generated transition tables, records an **append-only event log that is the receipt log**, and carries a trace ID on every command. Then prove by gate that the Kernel is the *only* thing in this repo that writes durable domain state.

## Context pack (read in this order, nothing else)

1. `START_HERE.md` — the One Rule.
2. `docs/BLUEPRINT.md` — L0/L1 rows, **Laws A–F** (E and F are enforced by this order), and the session-identity paragraph.
3. `docs/ONTOLOGY_SCHEMA.md` — §State machines: commands are rejectable intents, events are replayable facts, **there is no Receipt type**.
4. `qf-kernel-schema/` — you consume its generated output. You do not modify it.

## Deliverables

1. **`packages/qf-kernel/`** — a Bun + TypeScript package. The Kernel. No Electron, no canvas, no network, no model calls.

2. **Migration application.** The database schema comes from `qf-kernel-schema`'s generated `migration.sql`. **Do not hand-write DDL** and do not fork a copy — read the generated artifact. A hand-written table definition is a defect: schema drift is supposed to be impossible by construction.

3. **Command layer** — `execute(command, input, ctx)`:
   - validate the intent against the transition table via `assertTransition` from `qf-kernel-schema`;
   - on rejection, fail with a typed error naming type, from, and to — **and write nothing**;
   - on success, append an event, commit, and return the new state.
   Commands are rejectable; events are facts. One transaction per command.

4. **Event log = receipt log.** One append-only `events` table: `id`, `type` (dotted, e.g. `run.started`), `object_type`, `object_id`, `payload`, `trace_id`, `created_at`. **No `receipts` table, no Receipt object** — a parallel receipt store is a duplicate-truth Silo and fails this order.

5. **Trace context on every command.** `ctx` carries a `trace_id` (root per operation) and a `span_id`. Every event row records the `trace_id` that caused it. A command without trace context is rejected — make that a test.

6. **Session identity is adopted, never minted.** Per WO-004's finding (`docs/BLUEPRINT.md`): the ACP guest mints the session ID. `agent_session` rows take the supplied ID as their primary key. The Kernel must **reject** an `agent_session` insert with no supplied ID rather than generate one. Test both directions.

7. **Replay.** Given the event log alone, rebuild current object state and assert it equals the live table state. This is what makes the log authoritative rather than decorative. A one-object replay (`run` is enough) satisfies v0.

8. **Law E gates** — add to `qa/run.ts`, each failing loudly with the offending path:
   - **`kernel-sole-writer`** — no file outside `packages/qf-kernel/` opens a SQLite handle or issues DDL/DML for a domain type. Grep-based is acceptable for v0 (`bun:sqlite`, `better-sqlite3`, `CREATE TABLE`, `INSERT INTO`) with an explicit allowlist for `qf-kernel-schema` (generator output) and the Kernel itself.
   - **`no-canvas-domain-writes`** — no QuantFlow domain type is persisted through `canvas-state` / `canvas-persistence`.
   Both gates install their own dependencies and pass cold with no API key (see WO-004's gate for the pattern).

9. **Resolve the `commands.ts` drift — blocker from the WO-001→004 review.** `qf-kernel-schema/src/commands.ts` is a **third parallel catalog** that nothing imports. Measured: 18 verbs it invents are not schema actions (`grade_ticket_win`, `resolve_hypothesis_supported`, `settle_event`, …); 11 actions have no command (`close_run`, `retry_run`, `publish_artifact`, …); exactly **2 overlap** (`start_run`, `cancel_run`). Nothing lints the join, so it drifts silently — the exact Silo class WO-003 existed to prevent. Your command layer (deliverable 3) is what would consume it, so resolve it here. **Pick one and justify it in your report:**
   - **derive** commands from the transition tables and lint every command against a real schema action and a real legal transition; or
   - **delete** it, and build the command layer directly on `transitions` + `schema.actions`.
   Whichever you choose, a lint must make a fourth catalog impossible: any command name that is not a schema action, or any legal transition with no command, fails the build naming the offender.

10. **Spec defect to resolve — `close_run`.** `docs/ONTOLOGY_SCHEMA.md` lists `close_run` in the action surface, but `run` has no `closed` state (only `agent_session` does). **You may not edit the schema doc.** Report which of the two is right — remove the action, or add the state — with a one-paragraph recommendation. The architect lands the change by order; you implement `run` exactly as the transition table currently defines it.
