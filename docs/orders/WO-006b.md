# WO-006b — Kernel in the app: Artifact tile + the Law D cold-reopen demo
status: open
assignee: builder
depends: WO-006a

## Objective

The Electron app reads and writes domain truth only through the Kernel — proven by Law D: publish an Artifact, kill the app, relaunch, and the tile shows the same Artifact served from `kernel.db`.

## Context pack (read first, in order)

- `START_HERE.md` · `docs/BLUEPRINT.md` (Laws A–F; Law D is this order's acceptance, Law B its constraint)
- `packages/qf-kernel/src/index.ts` + `db.ts` + `execute.ts` — the API you are wiring in
- `collab-electron/src/main/index.ts`, `paths.ts`, `canvas-persistence.ts`, `ipc-canvas.ts`, and one preload file — the app idioms you must match
- `docs/orders/WO-006a.md` bottom verification records — what the Kernel now guarantees

**Measured facts this order relies on** (verifier, 2026-07-18, all measured inside the app's own Electron binary): Electron 40 bundles **Node 24.13** — `bun:sqlite` fails there (`ERR_UNSUPPORTED_ESM_URL_SCHEME`), but **`node:sqlite` (`DatabaseSync`) works built-in** (create/insert/select verified; emits an ExperimentalWarning, which is acceptable). **No new native dependency is needed or permitted.** `canvas-persistence.ts` stores tiles as layout JSON at `COLLAB_DIR/canvas-state.json` with types `term|note|code|image|graph|browser`; IPC idiom is `ipcMain.handle`.

## Deliverables

1. **Driver seam in `qf-kernel` (smallest possible).** Redefine `KernelDb` as an interface matching exactly what kernel code already calls — `query(sql): { run(...); get(...); all(...) }` and `transaction(fn)`. `openKernel` (bun:sqlite) still satisfies it; all existing tests keep running under Bun unchanged. Add `attachKernel(db: KernelDb): KernelDb` that runs the migration idempotently on an injected connection. No SQL changes, no behavior changes.
2. **`collab-electron/src/main/kernel.ts`** — the app's ONLY module importing `qf-kernel` (file: dependency) and the only module opening the DB: `node:sqlite`'s `DatabaseSync` at `COLLAB_DIR/kernel.db`, wrapped in a ~10-line adapter (`query(sql)` → `prepare(sql)`; both expose `run`/`get`/`all`) passed to `attachKernel`. No `better-sqlite3`, no native modules. Opened once at app-ready; log one line: `kernel: opened <path>, artifacts=<n>`.
3. **`collab-electron/src/main/ipc-kernel.ts`** — exactly two channels: `qf:execute` (command, input, trace → `execute()`; errors serialized with their names) and `qf:artifacts:list` (read-only SELECT via a new `listArtifacts(db)` helper exported from `qf-kernel`). Preload exposes them as `window.qf.execute` / `window.qf.listArtifacts`, matching the existing preload idiom.
4. **Tile type `"artifact"`** in `canvas-persistence.ts` TileState: stores `artifactId: string` and layout only — never kind, hash, or content. The renderer re-reads metadata from the Kernel on every mount. Canvas JSON stays a projection (Law D's point).
5. **Artifact tile window** (`windows/artifact-tile/`, cloned from the smallest existing tile window): renders id (short form), kind, `content_hash`, `storage_ref`, `created_at` — all fetched via `window.qf`, nothing cached to disk.
6. **The founder's demo lever:** an affordance (menu item or button — builder's choice, say which) that publishes a demo artifact: renderer sends a chosen file path (or fixed demo bytes), main reads the bytes, Kernel hashes and publishes, an artifact tile appears bound to the returned id.
7. **`docs/demos/law-d.md`** — the founder's script, exact steps: launch → publish → see tile → force-kill the app → relaunch → same tile content, plus one command that prints the `artifact.published` event row from `kernel.db` (the receipt).
8. **Gate: `kernel-sole-writer-app`** in `qa/` — static, falsifiable: fails if any file under `collab-electron/src` other than `main/kernel.ts` imports `qf-kernel` or any sqlite module, or if any file references `kernel.db` outside `kernel.ts`. Wire into `qa/run.ts --all`.

## Contract

- `qf-kernel` may gain ONLY: the `KernelDb` interface, `attachKernel`, `listArtifacts`. No schema, command, or SQL changes.
- Renderer processes never touch SQLite; domain writes cross exactly one seam (`qf:execute`); every call carries `trace_id`/`span_id` (renderer-generated UUIDs are fine for v0).
- No new state files. Canvas JSON = layout + references. Message caches and prefs are out of scope — do not migrate them.
- Match app idioms (module style, IPC registration, window creation) — this order adds a capability, not a refactor.
