# WO-006b — Kernel in the app: Artifact tile + the Law D cold-reopen demo

## Objective

The Electron app reads and writes domain truth only through the Kernel — proven by Law D: publish an Artifact, kill the app, relaunch, and the tile shows the same Artifact served from `kernel.db`.

- `START_HERE.md` · `docs/BLUEPRINT.md` (Laws A–F; Law D is this order's acceptance, Law B its constraint)
- `packages/qf-kernel/src/index.ts` + `db.ts` + `execute.ts` — the API you are wiring in

- `docs/orders/WO-006a.md` bottom verification records — what the Kernel now guarantees

**Measured facts this order relies on** (verifier, 2026-07-18, all measured inside the app's own Electron binary): Electron 40 bundles **Node 24.13** — `bun:sqlite` fails there (`ERR_UNSUPPORTED_ESM_URL_SCHEME`), but **`node:sqlite` (`DatabaseSync`) works built-in** (create/insert/select verified; emits an ExperimentalWarning, which is acceptable). **No new native dependency is needed or permitted.** `canvas-persistence.ts` stores tiles as layout JSON at `COLLAB_DIR/canvas-state.json` with types `term|note|code|image|graph|browser`; IPC idiom is `ipcMain.handle`.

## Deliverables

1. **Driver seam in `qf-kernel` (smallest possible).** Redefine `KernelDb` as an interface matching exactly what kernel code calls — `query(sql): { run(...); get(...); all(...) }`, `exec(sql)` (the migration path uses it), and `transaction(fn)`. `openKernel` (bun:sqlite) still satisfies it; all existing tests keep running under Bun unchanged. Add `attachKernel(db: KernelDb): KernelDb` that applies the migration + events DDL idempotently via `exec` on an injected connection.
   **Electron-safe entry (measured requirement):** the current barrel re-exports `db.ts`, whose top-level `import "bun:sqlite"` would crash Electron on load. Split the bun-only code (`openKernel`/`closeKernel`) into its own module and add a package subpath export — `qf-kernel/portable` — exposing `attachKernel`, `execute`, `listArtifacts`, `contentHash`, types, and everything else that does not touch `bun:sqlite`. The app imports ONLY from `qf-kernel/portable`. No SQL changes, no behavior changes.
2. **`collab-electron/src/main/kernel.ts`** — the app's ONLY module importing `qf-kernel/portable` (file: dependency) and the only module opening the DB: `node:sqlite`'s `DatabaseSync` at `COLLAB_DIR/kernel.db`, wrapped in an adapter passed to `attachKernel`: `query(sql)` → `prepare(sql)` (both expose `run`/`get`/`all`), `exec` passes through, and `transaction(fn)` is implemented as `exec("BEGIN IMMEDIATE")` / run `fn` / `exec("COMMIT")`, with `exec("ROLLBACK")` on throw and the error re-thrown (measured: `DatabaseSync` has `exec` but no `transaction`). This adapter is explicitly authorized — roughly 25 lines total. No `better-sqlite3`, no native modules. Opened once at app-ready; log one line: `kernel: opened <path>, artifacts=<n>`.
3. **`collab-electron/src/main/ipc-kernel.ts`** — exactly two channels: `qf:execute` (command, input, trace → `execute()`; errors serialized as `{ name, message }` so the renderer can distinguish typed Kernel errors) and `qf:artifacts:list` (read-only, via a new `listArtifacts(db)` helper exported from `qf-kernel/portable`, returning full artifact rows ordered by `created_at` DESC). Preload exposes them on the **existing `window.api` bridge** (measured idiom: `contextBridge.exposeInMainWorld("api", …)` in `preload/universal.ts`) as `api.qf.execute` / `api.qf.listArtifacts` — do not invent a new top-level global.
4. **Tile type `"artifact"`** in `canvas-persistence.ts` TileState: stores `artifactId: string` and layout only — never kind, hash, or content. The renderer re-reads metadata from the Kernel on every mount. Canvas JSON stays a projection (Law D's point).
5. **Artifact tile window** (`windows/artifact-tile/`, cloned from the smallest existing tile window): renders id (short form), kind, `content_hash`, `storage_ref`, `created_at` — all fetched via `api.qf`, nothing cached to disk. **Shell wiring is in scope for this tile only** (measured: a tile also needs its entry in `electron.vite.config.ts`, the shell view config / `shell:get-view-config` path, tile-manager registration, and the preload view list). Wire it exactly the way the smallest existing tile type is wired; touching those files for any other purpose is out of scope.
6. **The founder's demo lever:** an affordance (menu item or button — builder's choice, say which) that publishes a demo artifact: renderer sends a chosen file path, main reads the bytes, Kernel hashes and publishes with `kind: "report"` and `storage_ref` set to the source file's absolute path; an artifact tile appears bound to the returned id.
7. **`docs/demos/law-d.md`** — the founder's script, exact steps: launch → publish → see tile → force-kill the app → relaunch → same tile content, plus one copy-paste `bun -e` one-liner (run from the repo root, using `bun:sqlite`) that prints the `artifact.published` event row from `kernel.db` — the receipt.
8. **Gate: `kernel-sole-writer-app`** in `qa/` — static, falsifiable: fails if any file under `collab-electron/src` other than `main/kernel.ts` imports `qf-kernel` or any sqlite module, or if any file references `kernel.db` outside `kernel.ts`. Wire into `qa/run.ts --all`.

## Contract

- `qf-kernel` may gain ONLY: the `KernelDb` interface, `attachKernel`, `listArtifacts`, and the bun-only/portable module split with its `qf-kernel/portable` subpath export. No schema, command, or SQL changes.
- Renderer processes never touch SQLite; domain writes cross exactly one seam (`qf:execute`); every call carries `trace_id`/`span_id` (renderer-generated UUIDs are fine for v0).
- No new state files. Canvas JSON = layout + references. Message caches and prefs are out of scope — do not migrate them.
- Match app idioms (module style, IPC registration, window creation) — this order adds a capability, not a refactor.

### D1 — `no-canvas-domain-writes` is red; CI would reject this branch

```
no-canvas-domain-writes: domain types in canvas persistence:
  - collab-electron/src/windows/shell/src/canvas-state.js (domain type "artifact")
FAIL  no-canvas-domain-writes        →  bun qa/run.ts --all exits 1
```

**Measured before ranking it — the code is Law B compliant.** Canvas persistence gained exactly `type: "artifact"` (a tile-type discriminator), `artifactId?: string` (a reference), and a default tile size. **No domain fields**: no `kind`, no `content_hash`, no `storage_ref`, no content. That is precisely what deliverable 4 authorized, and storing a pointer is *how* a projection references truth (Law A).

So this is a **false positive in substance** — and it still blocks merge, because the gate cannot distinguish a reference from a truth-store, and CI is red either way.

**Attribution: architect order defect, not builder error.** Deliverable 4 authorized a change that predictably trips an existing Law E gate; the order neither authorized updating that gate nor listed it among the builder-run gates — and it *forbade* `qa --all`, so the builder could not have discovered it. The builder followed the order exactly. This surfaced precisely where the builder/verifier gate split intends it to.

**Required fix — make the gate precise, not permissive.** Do **not** simply allowlist the string `artifact`; that would retire a real protection. Flag domain **field** names in canvas persistence (`content_hash`, `storage_ref`, `kind`, `status`, `grade`, `claim`, `metrics`, `params`, `content_size`, …), while permitting:
- tile-type discriminator strings, and
- reference fields of the form `<domainType>Id` (e.g. `artifactId`, `runId`).

**Falsify the new behaviour both directions and paste all four outputs:** add `content_hash` to a canvas tile interface → **red**; remove → green. Confirm `artifactId` + `type: "artifact"` stay **green**. Then `bun qa/gates/no-canvas-domain-writes.ts` green and `git diff` empty.

**Scope of this rework: D1 only.** Change nothing else — everything above is accepted. Same branch, `wo-006b`.
